## `push.sh` — non-standard commit behavior

`./push.sh "<message>"` is **not** a normal `git add . && git commit && git push`. It iterates `git status --short` and creates **one commit per changed file**, auto-prefixing the message with a conventional-commit type chosen from path/extension heuristics:

- `*.sol` → `feat`
- `*test*` / `*spec*` → `test`
- `*.md` → `docs`
- `*.toml` / `*.json` / `*.yaml` / `*.yml` / `*.env*` / `*config*` → `chore`
- `*.css` / `*style*` → `style`
- deletions → `delete`, modifications fall back to `refactor`, additions to `feat`

Resulting messages look like `feat(src/Foo.sol): <message>`. Default message if omitted is `auto update`. Recent history (`feat(push.sh): auto update`, `docs(AGENTS.md): auto update`) was generated by this script — that pattern is intentional, not stale boilerplate to clean up.

Implication: if you stage a logical change spanning multiple files and run `push.sh`, you will get N separate commits rather than one. For grouped commits, use plain `git` directly.
